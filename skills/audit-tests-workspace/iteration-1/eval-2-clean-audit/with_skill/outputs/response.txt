### Top gaps
1. **Weak assertion on EUR formatting** — `toContain('100')` passes for any malformed output containing those digits. Should assert the full expected string.
2. **Missing failure path: invalid currency code** — `formatCurrency(100, 'INVALID')` throws a `RangeError`. If callers can pass arbitrary strings, this path is untested and could cause unhandled production errors.
3. **`clamp` with NaN / inverted range** — `clamp(NaN, 0, 10)` silently returns `NaN`; `clamp(5, 10, 0)` silently returns `0`. Both are surprising and untested.

### Detailed findings

**`formatCurrency`**

| Priority | Finding |
|----------|---------|
| P2 | EUR test uses `toContain('100')` — too weak. Assert the complete expected formatted string (e.g. `'€100.00'` or locale-equivalent). |

**`parseDate`**

| Priority | Finding |
|----------|---------|
| P2 | Timezone ambiguity: `'2024-01-15'` (bare date, no time/tz) is parsed as UTC midnight but `Date` behaviour can differ across environments. Either control `process.env.TZ` or document UTC-only inputs. |
| P2 | No epoch/far-future boundary tests (`'1970-01-01'`, `'2999-12-31'`). Low risk for this implementation, but worth a spot-check. |
| P3 | Three null/valid cases are good candidates for `test.each`. |

**`clamp`**

| Priority | Finding |
|----------|---------|
| P2 | No test for `NaN` input — `Math.min/Math.max` propagates `NaN`; callers won't know unless documented. |
| P2 | No test for inverted range (`min > max`) — current implementation returns `max` silently; behaviour should be tested and ideally documented. |
| P3 | Four parameterized-friendly cases — would be cleaner as `test.each`. |
