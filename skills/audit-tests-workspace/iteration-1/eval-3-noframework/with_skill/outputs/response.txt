### Top critical gaps

1. **`getProject` test is an empty body** ‚Äî the test exists, asserts nothing, and passes. Cross-tenant isolation (`row.tenantId !== user.tenantId` ‚Üí Forbidden) is completely unverified. A regression here would silently ship data to the wrong tenant.

2. **`deleteProject` is never imported or tested** ‚Äî this function contains the two most security-sensitive permission branches in the codebase (owner-only delete + admin override). Zero coverage on deletion means either branch could be broken and no test would catch it.

3. **`createProject` happy path is untested** ‚Äî the db stub always returns `[]`, so a successful create silently returns `undefined`. No test verifies that the returned project has the correct `tenantId` and `ownerId`, leaving the assignment logic unvalidated.

4. **Cross-tenant access via `getProject` is untested** ‚Äî a user from `tenantId: 't2'` requesting a project owned by `tenantId: 't1'` should be rejected; this is the core multi-tenant security invariant and there is no test for it.

### Detailed findings

#### P1 Critical

**1. `getProject` ‚Äî empty test body (vacuous pass)**
File: `src/projects.test.ts`, lines 14‚Äì17
The describe block exists with a comment acknowledging it is "fragile, not testing real persistence" ‚Äî but the body contains zero assertions. The test passes because it does nothing. Needs: a mock for `db.query`, then assertions for (a) project returned on valid same-tenant lookup, (b) 'Not found' thrown when query returns empty, (c) 'Forbidden' thrown when tenantId mismatches.

**2. `deleteProject` ‚Äî entirely absent**
File: `src/projects.ts`, lines 23‚Äì27 ‚Äî not imported in test file at all.
`deleteProject` enforces: only the owner OR an admin can delete. Neither branch is tested. This is domain-critical (data deletion + permission) and must have explicit tests.

**3. `createProject` ‚Äî success path untested**
File: `src/projects.test.ts` ‚Äî only the empty-name guard is tested. The happy path (db stub returns a row, function returns that row with correct shape) is uncovered. The current stub returns `[]` unconditionally, so any call to `createProject` with a valid name currently returns `undefined` and no test catches the bug.

**4. Cross-tenant isolation in `getProject` ‚Äî no test**
The `tenantId` guard (`row.tenantId !== user.tenantId`) is the core multi-tenant security control. There is no test where a user from a different tenant attempts to access a project, so a regression removing this guard would go undetected.

#### P2 Recommended

**5. `createProject` ‚Äî whitespace-only name not tested**
`'   '.trim()` is falsy, so the guard fires. Edge case of whitespace-only string should have its own test case.

**6. `getProject` ‚Äî 'Not found' path untested**
When `db.query` returns an empty array the function throws 'Not found'. No test covers this.

**7. `deleteProject` ‚Äî non-owner member deletion rejected**
A member who is not the project owner should be refused. No test exists.

**8. `deleteProject` ‚Äî admin can delete any project**
`user.role === 'admin'` bypasses the owner check. This special-case permission path needs an explicit test confirming the bypass works correctly.

**9. `deleteProject` ‚Äî delete of non-existent project propagates 'Not found'**
`deleteProject` calls `getProject` internally; if the project doesn't exist, 'Not found' should surface. Untested.

#### P3 Nice-to-have

**10. Dead test body with explanatory comment**
The `getProject` test body comment acknowledges the problem but the body is empty. Replace with a real test (using `vi.spyOn`) rather than a TODO comment.

**11. `db` is not injectable ‚Äî limits test control**
The `db` object is a module-level stub. Per-test return-value control requires `vi.spyOn(db, 'query').mockResolvedValueOnce(...)`. Consider exporting a `setDb` seam or accepting db as a parameter for cleaner testability.

**12. Test naming could be more behaviour-specific**
"throws when name is empty" is acceptable. Future tests should follow the same pattern: "rejects cross-tenant read access", "allows admin to delete another user's project", etc.
