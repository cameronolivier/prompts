## Test Audit: feature/billing-auth

### Overview
- Files analysed: 1 test file (`src/billing.test.ts`), 2 source files (`src/auth.ts`, `src/billing.ts`)
- Framework: Vitest (node environment)
- Test command: `npm test` / `vitest run`
- Untested source files: `src/auth.ts` (no test file at all)

### Findings

#### Unit Tests
- P1 Critical: 5
- P2 Recommended: 4
- P3 Nice-to-have: 3

#### Integration Tests
- None present; not flagged (no real DB or HTTP boundary on this branch)

---

### Top Critical Gaps

1. **`src/auth.ts` is entirely untested** — contains `checkPermission` (multi-tenant auth) and `revokeSession`. Auth and tenant isolation are always P1 per the rubric. A cross-tenant permission bypass would ship silently.

2. **`billing.test.ts` — `calculateInvoice` asserts only `toBeTruthy()`** — any non-null return passes. The subtotal/tax/total values are never verified. A wrong tax calculation would not be caught.

3. **`billing.test.ts` — `chargeCard` asserts only `toBeDefined()`; stripe is not mocked; `Math.random()` makes the test non-deterministic** — the test proves nothing about charge behaviour, and the `amount <= 0` guard and failed-charge error path are completely untested.

---

### Full Findings Detail

#### P1 Critical

**`src/billing.test.ts` — `calculateInvoice` weak assertion**
- Current: `expect(result).toBeTruthy()`
- Any object (including `{ subtotal: 0, tax: 0, total: 0 }`) passes this assertion.
- Needs: `expect(result.subtotal).toBe(20)`, `expect(result.tax).toBe(2)`, `expect(result.total).toBe(22)` for a `[{ price: 10, quantity: 2 }]` input.
- Why it matters: billing calculations must be exact. A 10x rounding error would pass this test.

**`src/billing.test.ts` — `chargeCard` weak assertion and non-determinism**
- Current: `expect(result).toBeDefined()` with no mock on the stripe client.
- `stripe.charge` uses `Math.random()` — different result on every run.
- Needs: mock `stripe.charge` to return a controlled fixture, then assert `result.id`, `result.status === 'succeeded'`, `result.amount`.
- Why it matters: test is non-deterministic (fails rubric §5) and asserts nothing specific about the charge shape.

**`src/billing.test.ts` — `chargeCard` missing validation failure path**
- The guard `if (amount <= 0) throw new Error('Amount must be positive')` has no test.
- Needs: a test that calls `chargeCard('cust_123', 0)` and asserts it rejects with the expected message.
- Why it matters: billing input validation must be verified explicitly.

**`src/billing.test.ts` — `chargeCard` missing failed-charge path**
- The path `if (result.status !== 'succeeded') throw new Error('Charge failed: ' + result.status)` has no test.
- Needs: mock stripe to return `{ status: 'failed', ... }` and assert the error is raised.
- Why it matters: partial charge failures would surface as unhandled exceptions without this coverage.

**`src/auth.ts` — entirely untested (no test file)**
- `checkPermission` enforces admin bypass, tenant isolation, and permission matching.
- `revokeSession` mutates in-memory session state.
- Both functions are domain-critical (auth, multi-tenancy) and completely without tests.
- Needs: a new `src/auth.test.ts` covering at minimum:
  - admin user returns true regardless of tenant/permissions
  - cross-tenant access returns false
  - correct tenant + matching permission returns true
  - correct tenant + missing permission returns false
  - null/undefined user throws
  - `revokeSession` removes the session

#### P2 Recommended

**`src/billing.test.ts` — `calculateInvoice` missing edge cases**
- Empty items list `[]` — what does it return? (likely `{ subtotal: 0, tax: 0, total: 0 }`, but untested)
- Zero-price item — subtotal 0, tax 0
- Floating-point rounding — e.g. `[{ price: 0.1, quantity: 3 }]` — does tax round correctly to 2dp?
- Single item vs. multiple items — currently only one case exists

**`src/billing.ts` — stripe boundary not mocked**
- `stripe.charge` should be replaced with a `vi.fn()` mock in all chargeCard tests.
- Without mocking, tests touch the simulated (but still non-deterministic) external boundary.

**`src/auth.ts` — `checkPermission` needs table-driven tests**
- Multiple role/tenant combinations are naturally expressed as parameterized cases.
- `it.each` over `[{role, tenantId, action, expected}, ...]` rows avoids copy-paste.

**`src/auth.ts` — `revokeSession` side-effect not verified**
- After calling `revokeSession(userId)`, nothing verifies the session is gone.
- Needs an integration-style test that sets a session then revokes it and checks absence.

#### P3 Nice-to-have

**Test names are vague**
- `'calculates invoice'` → `'returns correct subtotal, tax (10%), and total for multiple items'`
- `'charges card'` → `'returns charge id, status, and amount on successful charge'`

**`describe` block could be split**
- Single `describe('billing', ...)` block covers two unrelated functions.
- Split into `describe('calculateInvoice', ...)` and `describe('chargeCard', ...)`.

**Table-driven tests for `checkPermission`**
- Once auth tests are written, parameterize role/tenant/permission matrix with `it.each`.
